# 计划草稿页设计规格（Plan Draft Page）

日期：2026-04-09  
状态：与实现对齐（2026-04-10 全量验证通过）

## 1. 背景与目标

本规格用于定义“创建计划后进入草稿确认”的完整体验，确保用户在多个草稿版本中进行对比、选择并确认最终版本后，才进入正式执行流程。

核心目标（按优先级）：

1. 深度比较多个草稿版本，帮助用户做决策。
2. 通过明确确认动作（含二次确认）降低误操作。
3. 确认后立即进入正式详情（执行视图），形成闭环。

## 2. 硬性业务规则

1. 创建计划后先进入“计划草稿页”，而不是正式计划详情页。
2. 同一计划在草稿阶段最多 3 个版本（v1~v3），共用同一个 `planId`。
3. 用户点击“确认版本”时必须弹二次确认弹窗。
4. 用户确认某一版本后：
   - 该版本保存为正式计划；
   - 其他草稿版本立即删除，不作为历史保留。
5. 一旦计划成为正式计划（`active`），不能再回到草稿页。

## 3. 信息架构（方案 A：并排全景对照页）

### 3.1 顶部决策栏（固定）

- 左侧：计划标题、当前状态（草稿中/已确认）。
- 中间：版本计数（如 `v1 / v2 / v3`）。
- 右侧：
  - 全局“重新生成”按钮（基于当前选中卡片版本生成下一版）；
  - “确认当前选中版本”按钮。

**实现说明（当前前端）：** 顶部以副标题展示「状态：草稿待确认 · 已生成 N 个版本」；未单独放置居中的 `v1/v2/v3` 计数器，版本号在每张卡片标题与主按钮「确认 vX 并保存」中体现。重新生成按钮文案为「重新生成（剩余 M 次）」，与上限规则一致。

### 3.2 中部主区域：多卡并排对照

- 有几张草稿就并排显示几张卡片。
- 每张卡片包含：
  - 版本号与生成时间；
  - 每周/每时间段任务清单（全部展示，卡片内部滚动）；
  - 该版本摘要（简要说明）。
- 点击卡片即设为“当前选中版本”，并高亮样式。

**实现说明（当前前端）：** 版本摘要字段对应接口中的 `requirement`（界面文案为「版本说明」）；任务按 `stages` 分组展示。

### 3.3 底部差异分析区

- 比较粒度为：
  - 任务标题变化；
  - 所属阶段变化；
  - 预计时间变化。
- 支持查看选中版本与对照版本的差异统计与明细。

**实现说明（当前）：** 草稿页尚未挂载底部差异分析 UI；后端已提供 `GET /plans/:id/compare?base=&target=`（任务/阶段增删类 diff），后续可接入页面。产品层面对照与决策目标仍以 §1、§2 为准，不因当前未展示 UI 而弱化。

### 3.4 响应式策略

- 桌面端：多卡并排，容器允许横向滚动。
- 小屏端：单列展示 + 顶部版本切换器。

## 4. 关键交互流程

### 4.1 进入草稿页

- 从创建页点击“生成计划”后进入草稿页路由（建议 `/plans/:id/draft`）。
- 默认选中最新版本卡片。

**实现说明（当前前端）：** 若接口返回 `confirmedVersion` 则优先选中该版本，否则选中最新版本。

### 4.2 选中与重生成

- 点击任一草稿卡片即为当前选中版本。
- 点击顶部“重新生成”后，基于当前选中版本生成下一版。
- 达到 3 版上限时禁用重生成，并提示“已达版本上限”。

**实现说明（当前）：** 前端在无 `canRegenerate` 时禁用按钮；服务端在已达上限时返回 `409`，`message` 为 `version limit reached`，错误以 Toast 等形式展示。若计划已确认（主数据上已有确认版本），对 `POST /plans/:id/regenerate` 再调用会返回 `409`，`message` 为 `plan is already confirmed`（与 `GET /plans/:id/draft` 在 `active` 下的 `draft is closed` 区分）。

### 4.3 确认并提交

- 用户点击“确认 vX 并保存”后，打开二次确认弹窗。
- 二次确认通过后提交确认请求。
- 成功后直接跳转到该计划的正式详情页（执行视图）。

### 4.4 已确认访问草稿页

- 若计划已确认（`active`），访问草稿路由时不再展示草稿内容。
- 前端应直接重定向到正式详情页，并可附带一次性提示“该计划已确认，草稿已关闭”。

**实现说明（当前前端）：** 进入页时先 `GET /plans/:id`，若 `status === 'active'` 则 `replace` 至正式详情；若拉取草稿得到 `409` 且信息含 `draft is closed`，同样跳转详情。当前未展示一次性全局提示（可作为后续增强）。

## 5. 状态模型与数据约束

### 5.1 状态机

- `draft`：草稿阶段，可重生成，可确认。
- `active`：正式阶段，不可回退草稿。

状态迁移：`draft -> active`（仅确认动作触发）。

### 5.2 版本约束

- 版本号自增：`v1`、`v2`、`v3`。
- `planId` 不变。
- 确认后仅保留被确认版本在主计划数据中，删除其余草稿数据。

## 6. 接口契约建议

### 6.1 创建计划

- `POST /plans`
  - 结果：创建主计划（`status=draft`）并落地 `v1` 草稿。

### 6.2 重生成

- `POST /plans/:id/regenerate`
  - 输入：当前选中版本上下文（body 可含 `requirement`，用于基于选中快照覆盖/补充说明）。
  - 输出：新版本草稿（同 `planId`，`version+1`），以及 `versions`、`maxVersions`、`confirmedVersion`、`canRegenerate`。

### 6.3 确认版本（关键）

- `POST /plans/:id/confirm`
  - 输入：`version`。
  - 事务语义：
    1. 将该版本内容写回主计划；
    2. 主计划状态更新为 `active`；
    3. 删除该 `planId` 下除确认版本外的草稿版本。
  - 输出：正式计划摘要（用于前端跳转执行页）。

### 6.4 草稿读取

- `GET /plans/:id/draft`
  - 当计划为 `draft`：返回草稿版本集合。
  - 当计划为 `active`：返回 `409`，`message` 为 `draft is closed`（或等价语义），前端应跳转执行页。

### 6.5 版本差异（供对照扩展）

- `GET /plans/:id/compare?base=<n>&target=<m>`
  - 输出：基线版本与目标版本之间任务/阶段等结构化 diff（当前草稿页未消费，与 §3.3 一致）。

## 7. 错误处理与用户提示

- 重生成失败：保持当前卡片状态，展示可关闭错误提示，允许重试。
- 确认失败：弹窗不自动关闭，展示错误信息，允许再次确认。
- 并发确认冲突：提示“计划已被确认，草稿已关闭”，并跳转执行页。

## 8. 验收标准

1. 创建后进入草稿页，且可见 `v1`。
2. 最多可生成到 `v3`，超过上限不可继续。
3. 卡片支持并排展示所有草稿，卡片任务区可滚动。
4. 确认必须经过二次确认弹窗。
5. 确认某版本后，其他草稿版本被删除。
6. 确认成功后跳转执行页。
7. 计划进入 `active` 后无法回到草稿页。
8. 小屏端为单列 + 顶部版本切换器。
